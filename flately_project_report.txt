EXECUTIVE SUMMARY
Flately is a full-stack, real-time roommate-matching platform engineered to solve the practical housing pain point: roommate decisions are high-risk and usually made with low-quality context. Built on a robust technology stack (React 19, Node.js 22.x, Express, MongoDB, Socket.IO), Flately leverages advanced system design principles to ensure high performance, maintainability, and scalability. This report serves as a detailed dossier of the design patterns, architectural decisions, and collaborative team effort implemented in the Flately codebase.

2. ARCHITECTURAL OVERVIEW
--------------------------------------------------------------------------------
Flately adopts a clear, modular architecture following a Controller-Service-ORM pattern. The backend ensures strict single-responsibility boundaries for every feature group (Auth, Discovery, Matches, Chat, Preferences, Profiles).

CLIENT (Browser): React 19, Redux Toolkit, Vite 7, TailwindCSS v4, Socket.IO Client
BACKEND SERVER: Express 5, TypeScript, Socket.IO, Prisma ORM, JWT Middleware
DATABASE: MongoDB Atlas

Real-time communication for live chatting is handled via WebSockets (Socket.IO), which bypass standard REST polling overhead, providing an instant messaging experience.

--------------------------------------------------------------------------------
3. SOLID PRINCIPLES AT THE CORE
--------------------------------------------------------------------------------
To ensure long-term maintainability, our team enforced strict adherence to SOLID principles:
* Single Responsibility Principle (SRP): Feature groups are cleanly broken into Route -> Controller -> Service. Transports in the frontend isolate networking logic from UI components.
* Open/Closed Principle (OCP): New match eligibility parameters or scoring engines can be added by creating new Strategy classes without modifying the core MatchingService logic.
* Liskov Substitution Principle (LSP): Concrete profile and preference upsert strategies seamlessly substitute their abstract Template Method base (`UpsertByUserIdService`).
* Interface Segregation Principle (ISP): Small, highly focused interfaces like `EligibilityStrategy` and `ScoringStrategy` keep object dependencies minimal and isolated.
* Dependency Inversion Principle (DIP): High-level matching components depend solely on Strategy interfaces. Dependency injection bags are consistently passed into controllers and services.

--------------------------------------------------------------------------------
4. IMPLEMENTED DESIGN PATTERNS
--------------------------------------------------------------------------------
Flately relies on industry-standard Gang of Four (GoF) System Design patterns, verifying that our theoretical knowledge is properly applied to production-grade software:

A. CREATIONAL PATTERNS
* Factory Method: Used extensively in the Authentication flows. `AuthStrategyFactory`, `DefaultEmailAuthStrategyFactory`, and `DefaultOAuthAuthorizationStrategyFactory` centralize the creation of precise authentication strategies without tightly coupling the caller to specific configurations.
* Singleton: Applied broadly for system-wide module instances like Service Singletons, API Transport clients, and Socket connections. A lazy singleton Prisma client wrapper prevents premature database connections.

B. STRUCTURAL PATTERNS
* Adapter: The `HttpClientAdapter` adapts the frontend application's arbitrary network requests to standard Web Fetch APIs via `FetchRequestStrategy`. Domain-focused Transport classes adapt these raw API contracts into concrete business workflows.
* Proxy: Flately utilizes a JavaScript `Proxy` for `PrismaClient` to lazily instantiate the database client only upon the first actual property access, optimizing cold starts.
* Facade: Controllers and broad services export clean, unified API contracts while hiding complex downstream dependency orchestration internally (e.g., `DiscoveryService`, `CloudinaryService`).

C. BEHAVIORAL PATTERNS
* Strategy: Heavily anchors Flately's core algorithms.
  - `EligibilityStrategy` & `ScoringStrategy`: Drive the weighted recommendation engine.
  - `AuthContinuationStrategy`: Decides post-authentication routing routes dynamically (`QuestionnaireSourceStrategy`, `SignupDefaultStrategy`, etc.).
  - `RequestStrategy`: Abstracts the underlying network execution mechanism in the client.
* Template Method: The `UpsertByUserIdService` abstracts the invariant algorithm for sequence database upserts, allowing `ProfileUpsertService` and `PreferenceUpsertService` to define domain-specific variable steps like parameter mapping.
* Observer: Central to the realtime notification and messaging sub-system. Socket.IO Publisher-Subscriber events (`socket.on(...)`, `io.emit(...)`) combined with Redux state subscriptions create a robust reactive architecture.

--------------------------------------------------------------------------------
5. THE MATCHING ALGORITHM (RECOMMENDATION SYSTEM)
--------------------------------------------------------------------------------
Engineered by Ajeesh Amreet, the custom matching recommendation engine operates in two precise phases to ensure high-quality roommate suggestions:
1. Eligibility Filter (Hard Constraints): Validates non-negotiable criteria such as exact city matching, overlapping financial budget intervals, and mutual gender compatibility. 
2. Compatibility Scoring Engine: Calculates detailed similarity equations based on 1-5 scales (Cleanliness, Social Level, Sleep Schedule) and boolean data (Smoking, Drinking). This ensures deterministic sub-scores that bubble up to an aggregate 0-100% match probability index using deterministic tie-ordering.
